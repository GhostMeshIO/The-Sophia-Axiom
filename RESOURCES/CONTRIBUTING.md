## **Contribution Process**

### **Step 1: Preparation**
1. **Self-assess**: Ensure your coherence C > 0.60 (or working toward it)
2. **Study**: Thoroughly understand existing documentation
3. **Align**: Ensure your contribution aligns with framework principles
4. **Check**: Search for similar existing work to avoid duplication

### **Step 2: Development**
```
Fork → Branch → Develop → Test → Document
```

**Branch Naming Convention**:
- `doc/` - Documentation improvements
- `math/` - Mathematical developments
- `practice/` - Practical applications
- `research/` - Research contributions
- `community/` - Community building
- `fix/` - Bug fixes or corrections

**Example**: `doc/add-meditation-examples`, `math/prove-coherence-theorem`

### **Step 3: Submission**
1. **Update documentation** to reflect changes
2. **Write clear commit messages** explaining the "why"
3. **Reference related issues/discussions**
4. **Submit Pull Request** with:
   - Title: Clear description of contribution
   - Description: Purpose, changes, impact on coherence
   - Related issues: Links to any related discussions
   - Self-assessment: How you've validated the contribution

### **Step 4: Review Process**
1. **Automated checks**: Code/documentation formatting
2. **Community review**: Open for community feedback (7 days minimum)
3. **Core reviewer assessment**: Evaluation by framework maintainers
4. **Coherence impact assessment**: How contribution affects collective C
5. **Integration testing**: Ensure compatibility with existing framework

**Review Criteria**:
- **Clarity**: Is it understandable?
- **Coherence**: Does it fit with existing framework?
- **Innovation**: Does it add value (I > 2.45)?
- **Practicality**: Can it be implemented/applied?
- **Elegance**: Is it simple yet powerful (E > 50)?

## **Contribution Guidelines**

### **Documentation Standards**
- **Markdown formatting**: Use consistent headers, lists, code blocks
- **Mathematical notation**: Use LaTeX within `$$` for equations
- **Terminology**: Use established terms from Glossary.md
- **Citations**: Reference source materials appropriately
- **Accessibility**: Include alt text for images, clear structure

### **Code Standards**
```python
# Example: Coherence evolution function
def coherence_evolution(C_current, params):
    """
    Simulate one iteration of coherence evolution.

    Args:
        C_current: Current coherence (0-1)
        params: Dictionary of parameters

    Returns:
        C_next: Next coherence value

    Raises:
        ValueError: If parameters outside valid ranges
    """
    # Parameter validation
    validate_params(params)

    # Implementation with clear comments
    dC = compute_coherence_change(C_current, params)

    # Boundary enforcement
    C_next = np.clip(C_current + dC, 0, 1)

    return C_next
```

**Requirements**:
- Python 3.8+ compatibility
- NumPy/SciPy for mathematical functions
- Clear docstrings and type hints
- Unit tests for new functions
- No external dependencies without justification

### **Mathematical Standards**
- **Derivations**: Show steps, not just results
- **Notation**: Consistent with existing framework
- **Proofs**: Rigorous but accessible
- **Connections**: Relate to existing concepts
- **Validation**: Check against known results/limits
